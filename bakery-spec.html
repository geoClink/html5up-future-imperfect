The Bakery Co.

A full-stack bakery ordering and business management platform — customer storefront with real Stripe payments, a JWT-authenticated admin dashboard, transactional email and SMS, and scheduled cron jobs, all deployed on Render.

Live on Render Node.js Supabase Stripe HTML · CSS · JavaScript Resend Twilio
Real Stripe Payments
JWT Auth on All Admin Routes
3 Scheduled Cron Jobs

Demo

Site walkthrough — storefront, fuzzy search, cart, checkout, and admin dashboard.

Problem

Most portfolio bakery sites are static mockups — pretty but fake. The goal here was to build something that actually works end-to-end: a customer can browse the menu, add items to a cart, check out with a real card, and receive a confirmation email. A business owner can log into a dashboard, manage orders, update the menu, issue refunds, and export reports — without touching code.

Challenge-Based Learning

Challenge: Build a complete ordering platform from the ground up — real payments, a database, authenticated admin routes, email/SMS automation, and scheduled jobs — as someone still early in web development coming from iOS/Swift.

Approach: Used Node.js + Express for the backend, Supabase (Postgres) for the database, Stripe Elements for payments, Resend for transactional email, and Twilio for SMS. Deployed the whole stack to Render.

Outcome: A production-capable ordering platform with a real payment flow, a fully featured admin CMS, and automated operations running on scheduled cron jobs — the most technically complete web project I've built.

Key Features

Stripe Checkout

Real payment processing via Stripe Elements — tip selection, delivery fee calculation, and minimum order enforcement all handled before the card is charged. Webhooks update order status in Supabase after payment confirmation.

Admin Dashboard

Full order management, revenue tracking with Chart.js graphs, a menu CMS with image uploads, inventory management, one-click refunds via the Stripe API, CSV export, and thermal receipt printing — all behind JWT authentication.

Transactional Email & SMS

Branded HTML email templates sent via Resend for order confirmations, fulfillment notifications, and low stock alerts. Twilio SMS for real-time order status updates so customers don't have to check their inbox.

Scheduled Cron Jobs

Three automated jobs running on a schedule: a daily summary email, a weekly revenue report, and an unfulfilled order reminder — keeping the business informed without any manual effort.

Storefront & Cart

Live fuzzy search with keyboard navigation powered by Fuse.js. Shopping cart persisted to localStorage so items survive page refreshes. Customer order notes field so special requests reach the kitchen.

Security & Rate Limiting

JWT authentication (Supabase Auth) on every admin API route. API rate limiting via express-rate-limit to prevent abuse. No secrets exposed to the browser — all third-party API keys server-side only.

Challenges

Stripe Payment Flow — Tips, Fees, and Webhooks

The payment flow had more moving parts than a simple "charge card" integration. Tips are optional and percentage-based, delivery fees vary by zone, and orders below a minimum amount need to be blocked before Stripe is ever called. All of this had to be calculated server-side — trusting the client for cart totals is a security problem.

The Express backend computes the final charge amount and creates the Stripe PaymentIntent. After payment, a Stripe webhook fires and hits a /webhook endpoint that verifies the Stripe signature and writes the confirmed order to Supabase. The webhook — not the frontend — is the authoritative signal that money changed hands.

STRIPE WEBHOOK — server.js 
app.post('/webhook', express.raw({ type: 'application/json' }), async (req, res) => {
    const sig = req.headers['stripe-signature'];
    let event;

    try {
        event = stripe.webhooks.constructEvent(req.body, sig, process.env.STRIPE_WEBHOOK_SECRET);
    } catch (err) {
        return res.status(400).send(`Webhook Error: ${err.message}`);
    }

    if (event.type === 'payment_intent.succeeded') {
        const paymentIntent = event.data.object;
        const orderId = paymentIntent.metadata.order_id;

        await supabase
            .from('orders')
            .update({ status: 'paid', stripe_payment_id: paymentIntent.id })
            .eq('id', orderId);

        await sendOrderConfirmationEmail(orderId);
    }

    res.json({ received: true });
});

JWT Authentication on Admin Routes

The admin dashboard has real power — it can issue refunds, delete menu items, and export customer data. Every API route behind it needed to be protected so that only authenticated users could reach it.

Supabase Auth handles login and issues a JWT. On every admin API request, a middleware function verifies the token server-side before any data is read or written. An expired or missing token returns a 401 immediately.

AUTH MIDDLEWARE — middleware/auth.js 
const requireAuth = async (req, res, next) => {
    const authHeader = req.headers.authorization;
    if (!authHeader?.startsWith('Bearer ')) {
        return res.status(401).json({ error: 'No token provided' });
    }

    const token = authHeader.split(' ')[1];
    const { data: { user }, error } = await supabase.auth.getUser(token);

    if (error || !user) {
        return res.status(401).json({ error: 'Invalid or expired token' });
    }

    req.user = user;
    next();
};

// Applied to every admin route
app.use('/api/admin', requireAuth);

Scheduled Cron Jobs

The admin needed to receive a daily summary, a weekly revenue report, and alerts for orders that had been placed but not fulfilled — without having to remember to check anything manually. These needed to run on a schedule even when no one was using the app.

Three cron jobs run server-side using node-cron. Each one queries Supabase for the relevant data, formats it, and sends it via Resend. The unfulfilled order reminder only fires if there are actually open orders — no noise when the queue is clear.

CRON JOBS — jobs/scheduler.js 
import cron from 'node-cron';
import { sendDailySummary, sendWeeklyReport, sendUnfulfilledReminder } from './emailJobs.js';

// Daily summary — 8am every day
cron.schedule('0 8 * * *', sendDailySummary);

// Weekly report — Monday 9am
cron.schedule('0 9 * * 1', sendWeeklyReport);

// Unfulfilled order reminder — every 4 hours
cron.schedule('0 */4 * * *', sendUnfulfilledReminder);

Admin Refund Button — Hitting the Stripe API

The dashboard needed a refund button so the business owner could process returns without logging into Stripe. The challenge was making it idempotent — pressing the button twice shouldn't issue two refunds.

The refund endpoint checks whether the order already has a refund_id in Supabase before calling Stripe. If it does, it returns the existing refund instead of creating a new one. After a successful Stripe refund, the order status is updated to refunded so the dashboard reflects reality.

Outcome

The Bakery Co. is the most technically complete project I've shipped as a web developer. It covers the full stack — frontend UX, backend API design, relational database, real payments, transactional messaging, and automated operations. Every feature required understanding how the pieces connect, not just how to write code in isolation. Coming from iOS development, it gave me hands-on experience with the parts of web engineering that don't exist in the mobile world: server-side auth, webhook event handling, and scheduled background jobs.

What I Learned

  • How Stripe's payment flow works end-to-end — PaymentIntents, webhooks, and why the webhook is the only reliable signal that payment succeeded
  • JWT authentication on a Node.js/Express backend — issuing tokens, verifying them in middleware, and what 401 actually means
  • Supabase as a Postgres backend — relational data modeling, row-level security, and Supabase Auth
  • Transactional email with Resend — HTML email templates, deliverability basics, and when to use email vs. SMS
  • Scheduled cron jobs — running background work on a server without user interaction
  • Express API design — routing, middleware chains, rate limiting, and keeping secrets server-side
  • Deploying a Node.js app to Render with environment variable configuration
