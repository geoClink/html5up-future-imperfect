The Bakery Co.

A spec bakery website built from scratch with HTML, CSS, and vanilla JavaScript — complete with live fuzzy search, working forms, 40+ product pages, scroll animations, and a cart UI.

Live on GitHub Pages HTML CSS JavaScript Fuse.js Formspree GitHub Pages
40+ Product Pages

Demo

Site walkthrough — live fuzzy search, product pages, cart UI, forms, events, and employment page.

Problem

When learning web development, it's easy to build toy examples that never feel real. The goal of The Bakery Co. was to build something that actually functions like a business site — real navigation, real search, real form submissions, real product pages — using only the fundamentals. No React. No backend. No shortcuts. Just HTML, CSS, and enough JavaScript to make it feel alive.

Challenge-Based Learning

Challenge: Build a fully functioning business website that a real bakery could demo — with search, forms, dozens of products, and a consistent design — using only HTML, CSS, and vanilla JavaScript.

Approach: Integrated Fuse.js for live client-side fuzzy search, used Formspree to handle form submissions without a backend, hand-built 40+ product pages to simulate a real catalog, and deployed to GitHub Pages.

Outcome: A live, deployed spec site that demonstrates solid web fundamentals, third-party library integration, and the discipline required to maintain consistency across a large number of static pages.

Project Snapshot

  • Platform: Static site hosted on GitHub Pages
  • Stack: HTML · CSS · Vanilla JavaScript · Fuse.js · Formspree
  • Type: Spec project — designed and built as a learning exercise
  • Team: Solo
  • Role: Designer and developer — every page, every style, every script

Key Features

Live Fuzzy Search

Powered by Fuse.js. Users type into the search bar and results update instantly — no server, no page reload. Searches across product names and descriptions with fuzzy matching so minor typos still return results.

Working Forms

Contact, order, and employment forms all submit and deliver real emails via Formspree. No backend needed — Formspree handles delivery and spam filtering so the forms behave as if the site had a server.

40+ Product Pages

Every product has its own dedicated HTML page with consistent layout, images, descriptions, and pricing. Hardcoded for demo purposes but structured to show what a real product catalog would look like at scale.

Scroll Animations

Elements fade and slide into view as the user scrolls using the Intersection Observer API — no animation library required. Kept lightweight and purposeful so they add polish without slowing the page down.

Cart UI

A cart sidebar that tracks selected items and quantities, built in vanilla JavaScript. State lives in-memory and in the DOM — designed to show the full user flow of browsing, selecting, and reviewing an order.

Events & Employment Pages

Dedicated pages for bakery events and open positions, each with its own layout and a live Formspree form so visitors can RSVP or apply directly without leaving the site.

Challenges

Building Live Search Without a Backend

The first version of the search bar filtered a static JavaScript array but only matched exact substrings — so typing "choclate" returned nothing when it should have found the chocolate croissant. I needed fuzzy matching that ran entirely on the client.

I integrated Fuse.js via CDN with no build step. Product data lives in a flat JavaScript array. On every keystroke, Fuse scores and ranks matches based on edit distance and field weight. The threshold controls how forgiving the match is — 0 is exact, 1 matches almost anything. Setting it to 0.4 hit the right balance between precision and tolerance.

FUSE.JS SEARCH — script.js 
const fuse = new Fuse(menuItems, {
    keys: ["name", "description"],
    threshold: 0.4,
});

searchInput.addEventListener("input", () => {
    const query = searchInput.value.toLowerCase().trim();
    searchResults.innerHTML = "";
    activeIndex = -1;

    if (query === "") { searchResults.hidden = true; return; }

    const matches = fuse.search(query).map(result => result.item);

    if (matches.length === 0) {
        const li = document.createElement("li");
        li.innerHTML = `<span class="no-results">No results for "${searchInput.value}"</span>`;
        searchResults.appendChild(li);
        searchResults.hidden = false;
        return;
    }

    matches.forEach(item => {
        const li = document.createElement("li");
        li.innerHTML = `<a href="${item.href}">
            <span class="result-name">${item.name}</span>
            <span class="result-category">${item.category}</span>
        </a>`;
        searchResults.appendChild(li);
    });

    searchResults.hidden = false;
});

// Close dropdown when clicking outside
document.addEventListener("click", (e) => {
    if (!e.target.closest(".search-wrapper")) {
        searchResults.hidden = true;
    }
});

// Keyboard navigation (↑ ↓ Enter Escape)
searchInput.addEventListener("keydown", (e) => {
    const items = searchResults.querySelectorAll("li a");
    if (!items.length || searchResults.hidden) return;

    if (e.key === "ArrowDown") {
        e.preventDefault();
        activeIndex = (activeIndex + 1) % items.length;
    } else if (e.key === "ArrowUp") {
        e.preventDefault();
        activeIndex = (activeIndex - 1 + items.length) % items.length;
    } else if (e.key === "Enter" && activeIndex >= 0) {
        e.preventDefault();
        items[activeIndex].click();
    } else if (e.key === "Escape") {
        searchResults.hidden = true;
        activeIndex = -1;
    }

    items.forEach((item, i) =>
        item.parentElement.classList.toggle("active", i === activeIndex));
});

Scroll Animations — IntersectionObserver

Sections fade in as the user scrolls using the native IntersectionObserver API. No library needed — any element with the fade-in class starts invisible and gets a visible class added the moment it enters the viewport.

SCROLL ANIMATION — script.js 
const observer = new IntersectionObserver((entries) => {
    entries.forEach(entry => {
        if (entry.isIntersecting) {
            entry.target.classList.add("visible");
        }
    });
});

document.querySelectorAll(".fade-in").forEach(el => observer.observe(el));

The CSS does the rest — .fade-in starts at opacity: 0; transform: translateY(20px) and transitions to fully visible when .visible is added.

GitHub Pages Relative Path Issue

Everything worked perfectly when opening files locally. After deploying to GitHub Pages, images and stylesheets were broken on sub-pages. The browser was resolving relative paths like ../assets/css/main.css differently depending on the URL structure the deployment generated.

The fix was switching from relative paths to root-relative paths starting with /the-bakery-co/ consistently across every file. This made asset resolution predictable regardless of how deep a page sat in the directory tree. It was tedious to update — but it taught me exactly how the browser resolves paths and why it matters for any static site that isn't served from the root.

Updating 40+ HTML Files

Early on I added a new nav link and realized it meant opening and editing every single HTML file by hand. No templating engine, no shared components — just raw copy-paste across 40+ pages. After doing it twice I stopped and changed how I worked.

I started writing nav and footer markup in a way that was byte-for-byte identical across every file so a find-and-replace in VS Code could handle the whole site in one pass. It also made the case to me, viscerally, for why React and every other component-based framework exists. Shared components are not an abstraction for abstraction's sake — they solve a real, painful problem that you only fully understand after hitting it.

Outcome

The Bakery Co. is a working spec site that covers more ground than most beginner web projects. Building it without a framework meant confronting the browser directly — how paths resolve, how the DOM works, how to wire up a third-party library without npm. The challenges I ran into weren't problems to avoid; they were the whole point. Each one left me with a clearer mental model of how the web actually works.

What I Learned

  • How the browser resolves relative vs. root-relative paths, and why deployed sites break when you get it wrong
  • Integrating a third-party JavaScript library via CDN without npm, a build step, or a bundler
  • Handling form submissions without a backend using Formspree
  • The Intersection Observer API for performant scroll animations
  • Why shared components exist — learned by maintaining 40+ files without them first
  • Keeping a large static site consistent enough that global changes stay manageable
